# 2026-06-12 Round 276 Evaluation

## Focus
- Surface provider readiness problems inside the automation creation flow, not only in diagnostics.

## Changes
- Added an automation readiness card inside the "Profile & basic information" step.
- The card shows:
  - ready provider count,
  - missing or ready provider chips,
  - guidance to check token, Base URL, and OAuth callback before saving,
  - a direct "프로필/Callback 확인" action that opens the profile/integration tab.
- Added responsive CSS so the readiness card and action button stack cleanly on mobile.
- Treated an empty/not-yet-loaded provider readiness response as "needs checking" instead of incorrectly displaying the all-good state.
- Updated `verify:automation-form-ui` to assert:
  - the readiness card is visible,
  - provider status chips are rendered,
  - the profile/callback action opens the integrations panel.

## Current Result
- Users can now see integration readiness before saving an automation and can jump directly to profile/callback setup from the automation form.
- Existing automation sharing, profile setup, and automation form verifiers still pass against the public Cloudflare URL.

## Issues Found And Resolved
- Initial `verify:automation-form-ui` used a nonexistent `#profile-panel` selector.
- Resolved by checking the actual `#integrations-panel` target after clicking "프로필/Callback 확인".
- Review found that an empty provider-readiness response would make the new card look successful.
- Resolved by requiring a non-empty provider list before showing the "ready" state and adding a fallback warning chip while status is loading.

## Next Risks
- Provider readiness status should eventually become route-aware, so a GitHub -> Notion template highlights GitHub and Notion first instead of showing a generic provider list.
- Webhook registration remains external-console-dependent and should get more explicit per-provider setup state.
