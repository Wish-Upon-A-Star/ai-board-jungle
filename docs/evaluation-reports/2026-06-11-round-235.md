## Review Rounds
1. Read the latest Round 234 report and identified real provider-token verification as the next highest-risk gap.
2. Inspected `scripts/test-live-integrations.mjs`; found it only loaded `.env` and `backend/.env`.
3. Inspected desktop token filenames and key names without printing values.
4. Added desktop file loading for Codex/OpenAI, GitHub, Notion, Figma, and Google token files.

6. Added `--check-config` so readiness can be checked without external writes or token leakage.
7. Hardened raw-line parsing so Korean labels or comments are not mistaken for API tokens.
8. Added `owner/repo` GitHub repository parsing.
9. Added Notion page append fallback when the target is a Notion page instead of a database.
10. Added `--skip-missing` so valid configured providers can be tested even when another provider is incomplete.

## Result
- The live integration test no longer fails merely because it cannot see desktop token files.
- Current external-write blocker is now proven to be credential state:
  - GitHub/Notion token files are loaded, but the APIs reject them with 401.
  - Google/Figma token files do not currently contain usable access-token/file-target values for this verifier.
- App-mediated dry-run/write-gate verification still passes, so the application layer remains functional and safe; real external mutation needs valid provider credentials.
