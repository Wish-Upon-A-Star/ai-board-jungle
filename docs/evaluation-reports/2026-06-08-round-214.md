# Round 214 Evaluation

## Previous Reports Used

- Round 213: PostgreSQL-only runtime behavior was added, but live restart was intentionally deferred because no PostgreSQL server was reachable. The next priority was to make live application explicit, safe, and verifiable without taking down the current live server.

## Scope

- Added `scripts/apply-live.mjs` and `npm run apply:live`.
- The live apply command checks the configured PostgreSQL URL and TCP reachability before touching ports `3000` or `8000`.
- If PostgreSQL is unavailable, the command exits before stopping the current live web/API processes.
- If PostgreSQL is reachable, the command builds the frontend, restarts the live FastAPI and Vite services on `0.0.0.0`, and waits for both health checks.
- Documented the live apply procedure in `README.md`.
- Re-ran the user-style browser playtest against the currently running live server.

## Issues Found And Resolved

- Issue: There was no safe command to apply verified code to the live server without risking a restart into an unavailable PostgreSQL dependency.
  Resolution: Added `npm run apply:live`, which checks PostgreSQL first and exits before stopping live processes when the database is unavailable.
- Issue: The previous instructions could be misunderstood as keeping a test server updated while leaving live unchanged.
  Resolution: README now documents `apply:live` as the live promotion command after verification.
