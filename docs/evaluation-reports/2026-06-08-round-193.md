## Issues Found And Resolved

- Issue: Selecting a GitHub integration profile removed the Notion target from a GitHub-to-Notion automation.
  Resolution: Profile and form custom connections are now merged with de-duplication.
- Issue: Automation run could not actually write to Notion or create GitHub issues.
  Resolution: Added GitHub and Notion writer implementations and wired them into automation execution.
- Issue: GitHub commits/issues/PR changes did not affect the skip guard.
  Resolution: Newly saved external RAG source URLs are now included in the automation change hash.
- Issue: Webhook-based change detection was documented as future work.
  Resolution: Added `/api/webhooks/github` and `/api/webhooks/notion` with HMAC verification support.
- Issue: Automation save button could look like it did nothing.
  Resolution: Added saving/success/error state beside the button.

## Next Risks

- GitHub issue creation is implemented for configured create-issue operations; deciding what is a "problem" still depends on the user's automation instruction/template.
- Notion database schemas vary. The writer fills the title property and best-effort summary/link fields when matching properties exist.
- A stable production webhook URL still requires named Cloudflare Tunnel, reverse proxy, or hosted deployment.
