# Evaluation Round 01 - External RAG Collectors

## Scores

- Functionality: 8/10
- UI/design: 7/10
- Security/privacy: 7/10
- Performance: 7/10
- Tests: 8/10
- Documentation: 7/10
- Overall: 7.3/10

## What Improved

- Added real GitHub collector path for issues, commits, and pull requests when a user integration profile has a token.
- Added real Notion collector path for database pages and page blocks when a user integration profile has a token.
- Added `/api/integration-profiles/{id}/collect` to store collected external data into user-owned RAG knowledge sources.
- Added UI action `RAG 수집 실행` on each integration profile.
- Added test coverage for collector persistence through monkeypatched collector results.

## Remaining Risks

- Tokens are still stored as plaintext in the local DB. The API masks tokens in responses, but production should encrypt them or move them to a secret manager.
- Collector error display is functional but not yet polished for partial failures.
- Notion block extraction stores raw JSON; this is useful but not yet user-friendly.
- GitHub pagination is limited to the first page.
- UI still has dense forms; it is usable but can be visually cleaner.

## Next Improvement Targets

1. Add encrypted/token-secret abstraction or at least document local-dev-only plaintext storage.
2. Improve collector result UI with counts, warnings, and last run details.
3. Add pagination/limit controls for GitHub and Notion collectors.
4. Add a clearer RAG source dashboard grouped by source type.
5. Improve visual hierarchy of the long automation form.
